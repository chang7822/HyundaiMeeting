## 1. 배경 및 목적

### 현재 별(⭐) 시스템

| 획득 방법 | 지급량 |
|-----------|--------|
| 일일 출석 체크 | 1개 |
| 광고 시청 출석 | 2개 |
| 가위바위보 승리 | 배팅 × 2배 |

|--------|--------|

### 문제

자연 획득 속도(하루 최대 2개)에 비해 소비량(1회 10개)이 압도적으로 많아,  
적극적으로 서비스를 이용하려는 사용자에게 재화 부족이 빠르게 발생한다.  
유료 충전 경로를 열어 사용자 편의를 높이고 동시에 수익을 창출한다.

## 2. 결제 수단 선택: 가상계좌

### 왜 가상계좌인가?

- **카드 없이 결제 가능** — 앱/웹 사용자 중 모바일 간편결제 대신 계좌이체를 선호하는 직장인 다수
- **심사 없이 연동 가능** — 카드결제 대비 사업자 요건 완화
- **토스페이먼츠 무료 테스트 환경 제공** — 개발·검증 비용 없음
- **추후 카드, 간편결제로 확장 용이** — 동일 토스 SDK 사용

---

## 3. 전체 결제 플로우

```
[사용자 /shop 진입]
        ↓
[상품(별 패키지) 선택]  예: 30별 3,000원 / 100별 9,000원 / 200별 17,000원
        ↓
[은행 선택]  신한·국민·농협·우리·하나 등 13개 은행
        ↓
[프론트 → 백엔드 POST /api/payment/virtual-account]
        ↓
[백엔드 → 토스페이먼츠 API POST /v1/virtual-accounts]
        ↓
[토스 응답: 계좌번호, 은행, 입금기한, secret 수신]
  DB payment_orders에 저장 (status: WAITING_FOR_DEPOSIT)
        ↓
[프론트: 가상계좌 정보 표시 + 복사 버튼]
        ↓
[사용자 직접 입금]
        ↓
[토스페이먼츠 → 백엔드 Webhook POST /api/payment/webhook]
  이벤트: DEPOSIT_CALLBACK  body: { secret, status, orderId }
        ↓
[secret 검증 → status === "DONE" → 별 지급]
  awardStars(userId, stars, 'purchase', { orderId, amount })
        ↓
[DB 주문 status → DONE, deposited_at 기록]
        ↓
[인앱 알림 발송: "⭐ 30개가 충전되었습니다!"]
```

## 5. DB 스키마

### 5.1 `shop_products` — 판매 상품 목록

```sql
CREATE TABLE shop_products (
  id          SERIAL PRIMARY KEY,
  name        TEXT    NOT NULL,           -- "별 30개"
  stars       INT     NOT NULL,           -- 지급할 별 개수
  price       INT     NOT NULL,           -- 원화 가격 (KRW, 원 단위)
  bonus_stars INT     DEFAULT 0,          -- 보너스 별 (이벤트 프로모션용)
  is_active   BOOLEAN DEFAULT true,       -- 비활성화 시 상점에서 미노출
  sort_order  INT     DEFAULT 0,          -- 표시 순서
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 초기 데이터
INSERT INTO shop_products (name, stars, price, sort_order) VALUES
  ('별 30개',  30,  3000,  1),
  ('별 100개', 100, 9000,  2),
  ('별 200개', 200, 17000, 3),
  ('별 500개', 500, 40000, 4);
```

### 5.2 `payment_orders` — 결제 주문 내역

```sql
CREATE TABLE payment_orders (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          UUID REFERENCES users(id) NOT NULL,
  product_id       INT  REFERENCES shop_products(id) NOT NULL,

  -- 토스페이먼츠 식별자
  order_id         TEXT UNIQUE NOT NULL,  -- 우리가 생성하는 고유 주문번호 (UUID 기반)
  payment_key      TEXT,                  -- 토스 paymentKey (발급 후 채움)
  toss_secret      TEXT,                  -- 웹훅 검증용 secret (발급 후 채움, 보안)

  -- 결제 정보
  amount           INT  NOT NULL,         -- 결제금액 (원)
  stars_to_award   INT  NOT NULL,         -- 지급할 별 (bonus 포함)

  -- 상태
  -- PENDING: 발급 요청 전
  -- WAITING_FOR_DEPOSIT: 가상계좌 발급 완료, 입금 대기
  -- DONE: 입금 완료, 별 지급됨
  -- CANCELED: 취소됨 (기한 초과 등)
  -- FAILED: 발급 실패
  status           TEXT DEFAULT 'PENDING',

  -- 가상계좌 정보
  bank_code        TEXT,                  -- 은행 코드 (예: "신한")
  account_number   TEXT,                  -- 발급된 계좌번호
  customer_name    TEXT,                  -- 입금자명
  due_date         TIMESTAMPTZ,           -- 입금 기한

  -- 시각
  deposited_at     TIMESTAMPTZ,           -- 실제 입금 완료 시각
  created_at       TIMESTAMPTZ DEFAULT NOW(),
  updated_at       TIMESTAMPTZ DEFAULT NOW()
);

-- 조회 최적화 인덱스
CREATE INDEX idx_payment_orders_user_id ON payment_orders(user_id);
CREATE INDEX idx_payment_orders_order_id ON payment_orders(order_id);
CREATE INDEX idx_payment_orders_status   ON payment_orders(status);
```

## 6. 백엔드 구현

### 6.1 파일 구조

```
backend/
  routes/
    payment.js        ← 신규 생성
  server.js           ← 라우트 마운트 추가
```

### 6.2 환경 변수

`backend/config.env` 및 Render 백엔드 서비스 환경변수에 추가:

```env
TOSS_SECRET_KEY=test_sk_...    # 테스트키 → 라이브 전환 시 live_sk_... 로 교체
```

> `TOSS_SECRET_KEY`는 절대 프론트엔드 코드나 `.env` 파일에 노출하지 않는다.

### 6.3 엔드포인트 목록

| 메서드 | 경로 | 인증 | 설명 |
|--------|------|------|------|
| `GET`  | `/api/payment/products` | 필요 | 활성 상품 목록 조회 |
| `POST` | `/api/payment/virtual-account` | 필요 | 가상계좌 발급 요청 |
| `GET`  | `/api/payment/orders` | 필요 | 내 결제 내역 조회 |
| `POST` | `/api/payment/webhook` | **없음** | 토스 입금 웹훅 수신 |

### 6.4 가상계좌 발급 로직 (`POST /api/payment/virtual-account`)

```
1. req.body: { productId, bankCode, customerName }
2. DB에서 shop_products 조회 → amount, stars 확인 (프론트 금액 믿지 않음)
3. orderId = uuid() 생성
4. payment_orders INSERT (status: PENDING)
5. TossPayments API 호출:
   POST https://api.tosspayments.com/v1/virtual-accounts
   Authorization: Basic BASE64(TOSS_SECRET_KEY + ":")
   {
     orderId,
     orderName: "직쏠공 별 30개",
     amount: 3000,
     customerName: "홍길동",
     bank: "신한",          ← bankCode
     validHours: 24         ← 24시간 이내 입금
   }
6. 응답: paymentKey, virtualAccount.accountNumber, virtualAccount.dueDate, secret
7. payment_orders UPDATE: status → WAITING_FOR_DEPOSIT, 계좌정보 저장
8. 프론트에 계좌 정보 반환
```

### 6.5 웹훅 처리 로직 (`POST /api/payment/webhook`)

```
웹훅 body: { createdAt, secret, status, transactionKey, orderId }

1. orderId로 payment_orders 조회
2. body.secret === 주문의 toss_secret 비교 (불일치 시 400 반환)
3. 이미 DONE인 주문이면 중복 처리 방지 → 200 반환 후 종료
4. status === "DONE":
   a. awardStars(userId, stars_to_award, 'purchase', { orderId, amount })
   b. payment_orders UPDATE: status → DONE, deposited_at → NOW()
   c. 인앱 알림: notifications INSERT
   d. (선택) 푸시 알림 발송
5. status === "CANCELED":
   a. payment_orders UPDATE: status → CANCELED
6. 반드시 res.status(200).end() 반환 (10초 이내)
```

## 8. 보안 체크리스트

| 항목 | 처리 방법 |
|------|-----------|
| `toss_secret` 검증 | 웹훅 수신 시 DB와 반드시 비교, 불일치 시 무시 |
| 중복 지급 방지 | `status === 'DONE'` 이미 처리된 orderId는 skip |
| 금액 변조 방지 | 프론트에서 받은 금액 무시, 백엔드가 productId 기반으로 amount 결정 |
| `TOSS_SECRET_KEY` 보호 | 백엔드 환경변수로만 관리, 절대 프론트 노출 금지 |
| 웹훅 응답 속도 | 별 지급·알림은 비동기 처리, 웹훅 응답은 10초 이내 |
| PENDING 주문 만료 | 24시간 후 입금 안 된 주문은 스케줄러로 CANCELED 처리 (선택) |

## 9. 토스페이먼츠 개발자센터 설정

1. **회원가입** → https://developers.tosspayments.com
2. **내 앱 → API 키** 에서 테스트 Secret Key 확인
3. **웹훅 등록**: 개발자센터 → 내 앱 → 웹훅
   - URL: `https://auto-matching-way-backend.onrender.com/api/payment/webhook`
   - 이벤트: `DEPOSIT_CALLBACK` 체크
4. **테스트 입금**: 개발자센터 → 테스트 거래내역 → "입금처리" 버튼으로 웹훅 시뮬레이션
